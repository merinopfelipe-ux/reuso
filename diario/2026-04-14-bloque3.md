# Sesión 2026-04-14 — Bloque 3: Calculadora completa por kg + rich text

## Contexto de entrada

Continuación de [[2026-04-14-bloque2]] (Bloque 2: sidebar y dashboard bifurcado). Ahora se rediseña el corazón del producto: la calculadora pasa de input por unidades a input por kg, y se añade campo de descripción con soporte para pegar imágenes.

## Cambios implementados (Bloque 3)

### `src/lib/calculos/co2.ts` — nuevo modelo de cálculo por kg

**Antes:** `ItemCalculo.cantidad` (entero, unidades) × `co2_por_unidad` = CO₂  
**Después:** `ItemCalculo.peso_kg_input` (float, kg ingresados) × `factorCo2PorKg(co2_por_unidad, peso_kg_unidad)` = CO₂

Cambios:
- `cantidad: number` → `peso_kg_input: number` en `ItemCalculo`
- `peso_kg` renombrado a `peso_kg_unidad` (peso de UNA unidad, de BD)
- Nueva función exportada `factorCo2PorKg(co2_por_unidad, peso_kg_unidad)` que calcula el factor CO₂/kg
- Formula: `factor = co2_por_unidad / peso_kg_unidad` (si `peso_kg_unidad <= 0` retorna 0)
- `calcularImpacto` ahora usa `peso_kg_input * factor` en vez de `cantidad * co2_por_unidad`
- `factor_snapshot_json` incluye `co2_por_kg` precalculado para mayor trazabilidad

### `src/app/api/calcular/route.ts` — API actualizada

Cambios al schema Zod:
- `cantidad: z.number().int().min(1).max(1000)` → `peso_kg: z.number().positive().max(100000)`
- Añadido `descripcion_html: z.string().max(50000).optional()`

Cambios en `detalle_json`:
- Campo `cantidad` → `peso_kg` (peso ingresado por el usuario)
- Añadido `_descripcion_html` si fue enviado (prefijo `_` señala campo meta)

Cambios en `factor_snapshot_json`:
- Añadido `co2_por_kg` (precalculado en el snapshot para lectura directa sin recalcular)
- Añadido `peso_kg_unidad` (separado del peso ingresado por el usuario)
- Metodología: `'ACV simplificado, factores europeos ecoinvent/ELCD — input en kg'`

### `src/components/calculadora/calculadora.tsx` — UI reescrita

**Input por kg:**
- Estado `cantidades: Record<string, number>` → `pesos: Record<string, number>` (float)
- Input numérico con `type="number"`, `step={0.1}`, etiqueta "kg" al lado
- Badge del item muestra `X.XXX kg CO₂/kg` (factor por kg) en vez de `X.XX kg CO₂` por unidad
- Subtotal por item: `pesoInput × factorKg`

**Campo descripción con rich text + paste de imágenes:**
- `<div contentEditable>` con `ref={descripcionRef}`
- `onPaste` intercepta imágenes del clipboard: convierte el blob a base64 con `FileReader` e inserta `<img src="data:...">` via `document.execCommand('insertHTML')`
- Texto plano se pega normalmente sin intervención
- Placeholder implementado via `[contenteditable]:empty:before { content: attr(data-placeholder) }`
- Al guardar: `descripcionRef.current?.innerHTML` se envía como `descripcion_html`
- Al resetear: `descripcionRef.current.innerHTML = ''`

**Corrección de style tag:**
- `<style>{...}</style>` → `<style dangerouslySetInnerHTML={{ __html: ... }} />` (previene hydration mismatch)
- Ver [[hydration-style-dangerouslysetinnerhtml]]

## Decisiones técnicas

### Por qué kg y no unidades
El plan establece explícitamente "todo por PESO EN KG, nunca por unidad". Esto es más correcto ambientalmente — una camiseta de 100g y una de 500g no tienen el mismo impacto aunque ambas sean "1 unidad". El peso es la unidad universal que permite comparar diferentes categorías de material.

### factorCo2PorKg como función pura exportada
Se exporta desde `co2.ts` para que el componente cliente calcule los totales en tiempo real sin duplicar la fórmula. Principio DRY: una sola fuente de verdad para el factor.

### contentEditable vs `<textarea>`
`<textarea>` no puede contener HTML. Para rich text básico con imágenes embebidas se necesita `contentEditable`. La implementación es deliberadamente mínima — no se usa un editor de texto completo (Tiptap, Quill) porque el único caso de uso especial es el paste de imágenes. Texto plano y saltos de línea funcionan nativamente.

### document.execCommand('insertHTML') para paste de imágenes
`execCommand` está deprecado en el estándar pero sigue siendo la manera más práctica de insertar contenido en la posición del cursor en un contentEditable sin reescribir el manejo de selección manualmente. Todos los navegadores modernos lo soportan para este caso.

### Base64 en detalle_json
Las imágenes pegadas se guardan como strings base64 dentro de `_descripcion_html` en `detalle_json` (tipo JSONB en Supabase). Un blob de imagen típico pesa 20–200 KB en base64. Esto está dentro del rango razonable para JSONB y evita la complejidad de Storage por imagen individual.

### Selector de módulos temporalmente = selector de categorías
Hasta que el Bloque 8 añada la tabla `modulos`, las tabs de categorías actúan como "módulos". No se creó código intermedio innecesario.
