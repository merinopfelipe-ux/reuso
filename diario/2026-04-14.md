## Cambios implementados hoy (Bloque 0)

### Fix 1 — Ancho: eliminar maxWidth fijo

- **Problema:** `layout-shell.tsx` tenía `maxWidth: 1200` con `margin: 0 auto`. En pantallas >1200px el contenido quedaba encajonado con franjas blancas a los lados.
- **Solución:** Reemplazar por `width: '100%'` con solo `padding: '28px 32px'`. El contenido ahora ocupa toda el área disponible después del sidebar.
- **Archivo:** `src/components/layout-shell.tsx`
- Ver [[margin-auto-flex-vs-block]] para contexto del problema anterior con `margin: auto`.

### Fix 2 — Menú interno (PageSubmenu) a la derecha

- **Problema:** En `/settings`, el aside con el submenú estaba a la izquierda. El diseño del proyecto lo quiere a la derecha.
- **Solución en dos partes:**
  1. `page-submenu.tsx`: `borderLeft` → `borderRight`, `borderRadius: '0 8px 8px 0'` → `'8px 0 0 8px'`. También fix de `<style>` con `dangerouslySetInnerHTML`.
  2. `settings/page.tsx`: Invertir orden del flex — contenido primero, aside después.
- Ver [[page-submenu-derecha]] para el patrón.
- Ver [[hydration-style-dangerouslysetinnerhtml]] para el fix del `<style>`.

### Fix 3 — Nombre en el header: solo el primer nombre

- **Problema:** `HeaderUserDropdown` mostraba el nombre completo ("Luis Felipe Merino"), que se comprimía en el header.
- **Solución:** `nombre.split(' ')[0]` en el texto del trigger del dropdown. El avatar circular ya usaba `charAt(0)` — sin cambios.
- **Archivo:** `src/components/header-user-dropdown.tsx`

### Fix 4 — empresa_admin puede cambiar a modo colaborador

- **Problema:** El mapa funcional establece que empresa_admin puede conmutar a vista de empleado para usar la calculadora. No existía esta función.
- **Mecanismo:** Cookie `modo_empleado=1` con duración de 24 horas.
- **Implementación:**
  - `header-user-dropdown.tsx`: Nueva prop `rol: Rol`. Detecta cookie en `useEffect`. Muestra "Ver como colaborador" (empresa_admin) o "Volver a mi empresa" (en modo empleado).
  - `header.tsx`: Pasa `rol` al `HeaderUserDropdown`.
  - `(dashboard)/layout.tsx`: Lee cookie con `cookies()` de Next.js. Si `rol === 'empresa_admin'` y `modoEmpleado === true`, no redirige a `/empresa`. Usa `rolEfectivo = 'empleado'` para el `LayoutShell`.
- Ver [[modo-empleado-cookie]] para el patrón completo.

## Decisiones técnicas

### Ancho = 100% del workspace, no maxWidth
El sidebar ocupa su espacio fijo. El workspace es el resto. No hay razón para limitarlo adicionalmente — cada página decide su propio layout interno si necesita limitar el ancho (ej: formularios centrados).

### Cookie sobre localStorage para estado de sesión de rol
`cookies()` de Next.js está disponible en Server Components, lo que permite leer el estado en el layout y tomar la decisión de redirección sin round-trip al cliente. `localStorage` requeriría un Client Component, imposibilitando la guard en el layout.

### Primer nombre vs nombre completo en header
El header es espacio reducido. El nombre completo comprime otros elementos. El primer nombre es suficientemente identificador para el usuario que ya está logueado.

## Bloque 1 — Auth flows completos

### Cambios implementados (segunda parte de la sesión)

#### `/confirmar-email` — página nueva
- **Archivo:** `src/app/(auth)/confirmar-email/page.tsx`
- Input de 6 dígitos numéricos (`inputMode="numeric"`, auto-limpia no-dígitos, `maxLength=6`)
- Botón desactivado hasta que el código tenga 6 dígitos
- `supabase.auth.verifyOtp({ email, token, type: 'email' })`
- Botón "Reenviar código" con cooldown de 120s usando `setInterval` + `useRef` para limpiar
- `supabase.auth.resend({ type: 'signup', email })` para reenvío
- Estado de éxito: `CheckCircle` + auto-redirect a `/login` tras 2500ms
- **Patrón crítico:** Inner component `ConfirmarEmailContent` con `useSearchParams()`, envuelto en `<Suspense>` en la página exportada. Ver [[useSearchParams-suspense-nextjs]].

#### `/recuperar` — página nueva (2 pasos)
- **Archivo:** `src/app/(auth)/recuperar/page.tsx`
- **Paso 1 (`email`):** Input de correo → `supabase.auth.resetPasswordForEmail(email)` → siempre avanza al paso 2 (no revelar si el email existe; solo mostrar error en rate limit)
- **Paso 2 (`codigo`):** Input código 6 dígitos + nueva contraseña + confirmar → `supabase.auth.verifyOtp({ type: 'recovery' })` → `supabase.auth.updateUser({ password })` → `supabase.auth.signOut()`
- Validación client-side: min 8 chars, 1 mayúscula, 1 número, match de contraseñas
- Ícono ojo/cerrado para ambos campos de contraseña
- Estado de éxito: `CheckCircle` + auto-redirect a `/login`

#### `/registro/page.tsx` — ajuste post-registro
- Eliminada la pantalla estática "Revisa tu correo" (era placeholder con emoji ✓)
- Ahora: `router.push('/confirmar-email?email=' + encodeURIComponent(form.email))` tras registro exitoso
- El flujo queda conectado: registro → confirmar email → login

#### `src/middleware.ts` — rutas públicas
- Añadidos `/confirmar-email` y `/recuperar` a `PUBLIC_ROUTES`

### Decisiones técnicas del Bloque 1

#### Seguridad en recuperación: no revelar existencia de email
El paso 1 (`/recuperar`) siempre avanza al paso 2 independientemente de si Supabase responde con error para el email ingresado. Solo se muestra error si el mensaje contiene "rate limit" o "too many" — porque eso es un error técnico, no uno de seguridad. Ver [[password-recovery-security]].

#### OTP tipo 'recovery' vs tipo 'email'
Supabase usa tipos distintos: `type: 'email'` para confirmar cuenta nueva, `type: 'recovery'` para reset de contraseña. El token llega igual por email pero el tipo incorrecto resulta en error. Ver [[supabase-auth-otp]].

#### signOut después de cambiar contraseña
Supabase crea una sesión temporal cuando se verifica el OTP de recovery. Hay que hacer `signOut()` explícitamente después de `updateUser()` para que el usuario deba autenticarse de nuevo con la nueva contraseña.
